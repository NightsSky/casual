# Figma Prompts

## Prompt 1：整体重设计

```text
请基于这份产品说明、页面结构和组件状态描述，重新设计 casual 的移动端与桌面端界面。不要依赖截图。

产品定位：
casual 是一个 Flutter 开发的 Git 笔记应用，支持本地优先笔记、TXT/Markdown 编辑、GitHub/Gitee 同步和提醒助手。

目标用户：
个人开发者、技术写作者、Markdown 用户，以及习惯用 Git 管理内容的人。

设计目标：
1. 保留现有功能结构和业务状态，不新增无关功能。
2. 强化开发者工具感、清晰度和信息密度。
3. 移动端优先保证单手操作和可读性。
4. 桌面端强化侧边栏、列表、编辑器三者的效率。
5. 优化笔记列表、编辑器、仓库同步、设置表单和提醒弹窗。
6. 建立统一组件系统：导航、按钮、输入框、列表项、状态标签、工具条、弹窗、空状态。
7. 色彩要克制专业，避免单一暖棕色铺满全局。可以保留当前主色作为强调色，但请加入中性色和更清晰的状态色。
8. 不要做营销页，不要加入装饰性插画，不要把业务界面做成 landing page。

输出内容：
- Mobile：390x844 的核心页面设计。
- Desktop：1440x900 的核心页面设计。
- 组件规范：颜色、字号、间距、圆角、按钮、输入框、卡片、列表项、状态标签。
- 交互状态：空状态、加载、成功、失败、冲突、只读、删除确认。
```

## Prompt 2：只重设计笔记与编辑器

```text
请只重设计 casual 的笔记列表和编辑器。

必须保留：
- 笔记列表：标题、摘要、标签、更新时间、同步状态。
- 筛选：标签筛选。
- 操作：新建 TXT / Markdown、排序、同步、删除、桌面端在新窗口打开。
- 编辑器：标题、标签、正文、Markdown 工具条、编辑/预览切换、字数和更新时间。
- 状态：本地未同步、冲突、同步失败、独立窗口只读。

移动端：
- 笔记列表使用底部导航。
- 新建入口使用 FAB，展开后选择 TXT 或 Markdown。
- 编辑器隐藏底部导航，使用顶部返回。

桌面端：
- 左侧为 240px 侧边导航。
- 笔记模块使用分栏：列表 + 编辑器。
- 编辑器阅读模式要像适合长文阅读的文档区域，不要让行宽过长。

风格：
专业、清爽、开发者工具感。保持高可读性和稳定布局。
```

## Prompt 3：仓库同步与设置页

```text
请重设计 casual 的仓库管理页和设置页。

仓库管理页需要表现：
- GitHub/Gitee 仓库连接状态。
- 未配置、未连接、已连接三种状态。
- 拉取远程、推送本地、全量同步、仓库设置四个快捷操作。
- 总笔记、未同步、已同步、冲突统计。
- 同步记录列表，区分成功、失败、警告。

设置页需要表现：
- Git 平台选择：GitHub / Gitee。
- Access Token 隐藏输入。
- 用户名/组织、仓库名、分支、笔记目录。
- 如何获取 Access Token 的帮助入口。
- 测试连接、保存配置。
- 自动同步、编辑时自动推送开关。
- Windows 桌面关闭行为设置。
- 清除所有数据的危险操作确认。

设计目标：
减少表单疲劳，增强配置步骤感和错误修复提示，让第一次配置 Git 仓库的人也能完成。
```

## Prompt 4：提醒助手

```text
请重设计 casual 的提醒助手界面。

必须覆盖：
- 提醒列表：标题、时间或间隔、重复规则、启用开关。
- 空状态。
- 新建提醒弹窗。
- 编辑提醒弹窗。
- 重复方式：无、每天、每周、每月、工作日、按间隔。
- 间隔提醒：小时 + 分钟选择，保存后开始计时。
- 删除提醒确认。

设计方向：
提醒是一个辅助功能，不要抢占笔记主功能的视觉权重。界面应清晰、轻量、可靠，重点突出下一次提醒和启用状态。
```

## Prompt 5：设计评审

```text
请检查这套重设计稿是否完整覆盖 casual 当前 Flutter App 的功能。

请逐项检查：
1. 所有路由页面是否都有设计。
2. 移动端和桌面端是否都有对应布局。
3. 笔记同步状态是否完整：本地、已同步、冲突、同步中、同步失败。
4. 编辑器是否覆盖编辑、预览、Markdown 工具条、TXT/Markdown 格式转换。
5. 设置页是否覆盖 GitHub/Gitee、Token、测试连接、保存配置和危险操作。
6. 提醒是否覆盖单次、周期、间隔和删除确认。
7. 空状态、错误状态、加载状态是否存在。
8. 最终组件是否能转成 Flutter 组件，是否避免了过度装饰和难实现的布局。

请输出缺失项清单和修改建议。
```
